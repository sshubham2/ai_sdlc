# AI SDLC — Hybrid Pipeline

Spec-driven SDLC for AI implementers. Hybrid of phase-driven rigor (for knowable domains) and slice-driven iteration (for unknowable ones).

Each skill is a **drop-in Claude Code skill**. Copy any `skills/<name>/` directory into `~/.claude/skills/` and it works.

## Why this exists

Comprehensive-upfront-spec SDLC optimizes for **"never build the wrong thing."** That works for enterprise / compliance / large teams.

It falters when:
- The spec itself is the unknown (product discovery, consumer apps)
- Rework is cheap but late discovery is expensive
- Implementer is AI that benefits from plan-mode groundedness over 500-line sprint files

This pipeline keeps the durable parts of spec-driven work (vault, spikes, verification gates) and adds:

- **Risk-first slicing** — build the scary thing first
- **Two AI personas** — Builder produces, Critic adversarially reviews
- **Three modes** — Minimal / Standard / Heavy, picked by `/triage`
- **Mission brief over sprint file** — 1 page persistent + plan-mode execution
- **Continuous validation** — per slice, not terminal
- **Drift-check as pre-commit** — vault-code sync enforced automatically
- **Thin vault** — only what code can't carry (decisions, risks, slice memory). Components / contracts / schemas / test-plan / frontend live in code. Cure for spec rot at the root.
- **Project CLAUDE.md** — auto-generated by `/triage` to keep Claude on the pipeline across sessions (prevents the "ditch the pipeline and just start coding" failure)
- **Graphify throughout** — multi-modal knowledge graph (code + docs + papers + videos) auto-installed by `/triage`/`/adopt`; PreToolUse hook nudges Claude toward `GRAPH_REPORT.md` before Glob/Grep, and skills query the graph for reachability/blast-radius/path instead of re-scanning files. See [graphify-integration.md](graphify-integration.md).
- **milestone.md per slice** — rolling state file updated by every skill. Stage, next-action, progress, on-resume data all in one place. `/status` reads it as primary source. Survives session death, context clear, model confusion.
- **Risk-tier Critic gating** — `/slice` assigns tier (low/medium/high); `/critique` can skip low-tier slices that don't touch auth / contracts / data model / multi-device / external integrations. No more Critic reviewing CSS tweaks.
- **Honest Heavy mode** — `/heavy-architect` produces only human-authored files (threat model, requirements, NFRs). Code-derived views (components/contracts/schemas) generated on-demand by `/sync` before audits. Thin Vault principle preserved in all modes.

## Five principles

1. Iterate per slice, not per phase
2. Risk-first ordering
3. Two AI personas (Builder + Critic)
4. Reversibility-tagged decisions
5. Vault as commit metadata

See [principles.md](principles.md) for rationale.

## Pipeline at a glance

```
/triage                                      ← open the project, pick mode
  ↓
/discover (with /risk-spike, /user-test)     ← loop until slice 1 is clear
  ↓
[per slice:]
  /slice → /design-slice → /critique → /build-slice → /validate-slice → /reflect
  ↓ next slice
[maintenance: /drift-check, /reduce]
```

See [pipeline.md](pipeline.md) for the full spec.

## Setup (one-time per machine)

If you're using this pipeline, you already have Claude Code. Use it to install:

> **`cd` into the AI SDLC source directory (wherever you cloned/downloaded it), open Claude Code, and say:**
>
> "Install this. Read `INSTALL.md` and follow it."

Claude reads [INSTALL.md](INSTALL.md), detects what's on your machine, asks before mutating shared files (global CLAUDE.md, settings.json), executes the install, and verifies at the end. Idempotent — safe to re-run if you already ran it.

The install does:

1. Verifies / creates `~/.claude/.venv` (Python 3.11+)
2. Installs `graphify` (editable from `~/.claude/packages/graphify` if present, else PyPI)
3. Registers the graphify Claude Code skill at `~/.claude/skills/graphify/`
4. Adds the `PY=...` shared-Python convention to `~/.claude/CLAUDE.md` (with your confirmation)
5. Adds `CLAUDE_CODE_FORK_SUBAGENT=1` to `~/.claude/settings.json` (so `/fork` works)
6. Copies AI SDLC skills + agents into `~/.claude/skills/` and `~/.claude/agents/`

After setup: **restart Claude Code** so the env var loads. Then `/triage` and `/adopt` run a preflight check on first use; if anything is missing, they fail fast and point you back here.

## Getting started

If you're new: read **[tutorial.md](tutorial.md)** first. It has worked examples for each mode, decision tree, common scenarios, anti-patterns, and FAQ.

After installing the skills, in your project root:

```
/triage "what you're building in one sentence"
```

`/triage` picks a mode, creates `architecture/triage.md` + project CLAUDE.md, and tells you what to run next.

## Skill list

|-------|-------|-------|---------|
| `/triage` | Open | all | **Greenfield**: pick mode, build risk register, scope pipeline, generate project CLAUDE.md |

| `/diagnose` | Brownfield analysis | all | **Forensic, code-only**, owner-facing HTML deliverable. 11 analysis passes + narrative executive summary (via `diagnose-narrator` agent). Owner annotates findings inline; output feeds `/slice-candidates`. Used by `/adopt` Step 3 |
| `/slice-candidates` | Brownfield analysis | all | Reads annotated `diagnosis.html`, produces dependency-ordered `backlog.md` of slice candidates. Used by `/adopt` after the owner annotates the diagnosis |
| `/discover` | Discovery | all | Concept + users + constraints |
| `/risk-spike` | Discovery | all | Validate risky assumptions early |
| `/heavy-architect` | Discovery | **Heavy only** | Upfront comprehensive vault — components, contracts, schemas, threat model, cost estimation |
| `/user-test` | Discovery | Standard, Heavy | Real user validation gate (B2C) |
| `/slice` | Slice | all | Define next thinnest valuable cut |
| `/design-slice` | Slice | all | Just-enough spec for current slice |
| `/critique` | Slice | Standard, Heavy | Adversarial review by Critic persona |
| `/build-slice` | Build | all | Plan mode + verification gates |
| `/validate-slice` | Build | all | Real device / user / data check |
| `/reflect` | Build | all | Capture learnings, update vault |
| `/drift-check` | Maintenance | all | Vault vs code drift detection (pre-commit ready) |
| `/sync` | Maintenance | **Heavy only** | Bidirectional vault-code reconciliation; regenerates code-derived files |
| `/reduce` | Maintenance | all | Complexity budget enforcer (mandatory every 5 slices in Heavy) |

| `/commit-slice` | Finalize | all | Generate audit-grade commit message from mission-brief / build-log / validation / ADRs. No hand-crafting |

## The three modes

**Minimal** — solo dev / MVP / one-off scripts:
- Pipeline: `/triage → /discover → /slice → /design-slice → /build-slice → /validate-slice → /reflect → loop`
- Skips: `/critique`, `/user-test`, `/heavy-architect`, `/sync`
- Vault: thin (triage, concept, decisions, risks, slices)

**Standard** — B2C / small teams / product work (the default):
- Pipeline: `/triage → /discover → (/risk-spike if HIGH risks) → (/user-test if B2C) → /slice → /design-slice → /critique → /build-slice → /validate-slice → /reflect → loop`
- Maintenance: `/drift-check` (pre-commit), `/reduce` (when complexity grows)
- Vault: thin

**Heavy** — compliance / regulated / audit-grade:
- Pipeline: `/triage → /discover (full role-play) → /risk-spike all → /heavy-architect → /user-test → /slice → /design-slice (updates comprehensive vault inline) → /critique (mandatory + human sign-off) → /build-slice (compliance trail) → /validate-slice (reproducible, signed) → /reflect → loop`
- Maintenance: `/drift-check` (pre-commit), `/sync` (every 5-10 slices), `/reduce` (mandatory every 5 slices)
- Vault: comprehensive (components, contracts, schemas, threat model, cost estimation, requirements, NFRs, diagrams)

## In one sentence

**Triage first, slice small, build the scary thing early, let Critic challenge the Builder, make reality validate the vault, reflect every slice.**
