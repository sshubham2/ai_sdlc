AI Can Build Fully Engineered Software — You're Just Not Asking It Right

Maybe you've seen the vibe-coded badly-built house meme. GVM — the Grounded Vibe Methodology — is the fully engineered build: structure, review, quality gates, at vibe coding speed.

Conor Brennan & Gerard Quinn

The premise

GVM grounds AI in named, published experts at every phase of the software development life cycle — Tufte for data presentation, Wiegers for requirements, Fowler for refactoring, and so on. Naming the expert in the prompt focuses the model on a specific body of vetted work rather than a broad blend of everything in its training data. The rest of this post backs that premise two ways: we built software with it, and we tested the mechanism that makes it work.

We back the claim two ways. We built engineered software with AI — the case study below reports a tool built end-to-end with the methodology, and expert design patterns are visible in the production code, traceable line-by-line to the named experts active during design. We also tested the underlying mechanism — whether naming experts in the prompt changes what the model produces — in a controlled experiment, using dashboard generation as a measurable proxy task.

We ran the experiment across three AI model families — Claude, GPT, Gemini — under three grounding conditions per model. 180 dashboards. 1,080 blind reviews. The experiment tests the same mechanism that operates during system design, code review, and document writing and review. We chose the dashboard task because reviewers can score it against named experts' published criteria — Tufte on data-ink ratio, Few on alert design, Doumont on message structure — without knowing which grounding condition produced each dashboard. Dashboards are not what GVM builds; they are what we could measure cleanly.

The expected finding landed: more expert grounding produced measurably better output, with large effect sizes consistent across all three models.

The unexpected finding is the one worth the post. AI reviewers without expert grounding rated the better output worse. Independent inspection confirmed the grounded output was dramatically better. AI reviewers without the expert framework scored it below the baseline. They couldn't see the improvement because they didn't have the framework to recognise it.

The implication is uncomfortable. Most AI code review today is unguided AI review — and the experiment shows unguided reviewers cannot reliably distinguish better output from worse. The risk is that current AI code review may be approving work an expert-informed review would reject, and unable to detect that gap from inside its own framework. Three completely different AI systems from three different companies produced the same result; this isn't a quirk of one model.

The key finding

You can't see what you don't have a framework to see. Unguided AI review may be approving work that expert-informed review would reject — and the unguided reviewer cannot detect that gap from inside its own framework.

Two problems, not one

Most teams in 2026 have some kind of AI development process. The mainstream frameworks — spec-driven scaffolding, acceptance-test-driven workflows, walking-skeleton patterns — address pieces of the problem. None yet ships the combination of calibrated review that improves across rounds, build-time wiring audits, per-chunk smoke gates, and a release phase that reads its own verdict data. AI can write fluent, plausible, wrong code at unprecedented speed, and the quality gates have to match.

Problem one: structure. Requirements change and nothing tracks it. The AI's context fills up and the next session starts from zero. AI reviews treat every round as independent — the tenth review discovers the same issues as the first. There's no gate between requirements and architecture to check whether the requirements are even testable.

Problem two: quality. Even if you have structure, who defines what "good" looks like? A review that calibrates over time needs something to calibrate against. A test gate needs criteria. Without a quality framework, your process catches failures but converges on mediocrity.

You need both. Structure without quality standards catches bugs but ships mediocre work. Quality standards without structure produce beautiful code that nobody verified.

How GVM solves this

GVM — the Grounded Vibe Methodology — is a methodology designed specifically for building well-engineered software with AI. It runs from requirements through release. Two ideas define it.

The first idea: every phase has a verification mechanism. Requirements must prove they're testable before any architecture exists.

AI reviews track what they found across rounds and get more precise over time. From round 3, a second AI reviewer runs in parallel with no knowledge of prior rounds — seeing the project fresh to catch what the first reviewer's familiarity with the project history causes it to overlook. By that point, the first reviewer has seen enough history to develop expectations that create blind spots.

The release gate reads every review verdict and blocks if anything upstream isn't ready. If the documentation doesn't match the build, the release is blocked.

Review and release verify what was built; construction gates verify what is being built, catching a class of failure that review cannot reach because the broken artefact never reaches review intact. Tests must mock at the external boundary, not at internal collaborators. Every built module must name its consumer in a wiring matrix the build phase greps mechanically. Each chunk passes an end-to-end smoke check before it is accepted. These catch the wiring failures and the tests-pass-product-broken failures that would otherwise surface at integration time — or ship undetected.

The second idea: citation-based expert grounding. When you tell an AI "write good code," it draws from everything in its training data — a broad, undifferentiated blend. When you tell it "apply Fowler's refactoring principles from Refactoring, 2nd edition," it shifts to a focused application of a specific body of published work. The citation acts as a retrieval key into knowledge the model already has. No RAG. No fine-tuning. No vector database.

You might dismiss this as just better prompting. It is prompting — but the difference between "follow best practices" and "apply Wiegers' elicitation framework from Software Requirements" is the difference between telling a student to "do research" and telling them to read a specific chapter and apply a specific framework. One produces generic output. The other produces output grounded in a vetted methodology. You don't need to know which experts to cite — GVM selects the right experts for each phase automatically based on your project's domain and technology stack, and presents them for your approval before proceeding.

Build the right thing, then build the thing right

Software engineering has long been framed around two questions, not one — the verification/validation split articulated in Boehm's 1981 work and standard in the V&V literature ever since. Are we building the right thing? — and — are we building the thing right? The construction-side gates above answer the second. The first is just as easy to get wrong with AI: a fluent codebase that solves the wrong problem still solves the wrong problem.

GVM's discovery side covers the first question structurally. An optional impact map opens the pipeline, tracing goals to actors to deliverables before any requirements exist. Requirements are written acceptance-criteria-first — if you cannot say how you would verify it, the requirement is not yet finished. Test cases are generated before architecture, so a requirement that is plausible-but-untestable is caught at the requirements phase, not after the AI has built around it. A walking skeleton wires the real (or sandbox) integration boundaries end-to-end before feature work starts; if the skeleton goes red, feature work is refused. A timeboxed exploratory testing pass against the running product catches what scripted tests miss. Each of these is grounded in named experts: Adzic on impact mapping, Wiegers and Patton on requirements, Cockburn and Freeman & Pryce on the walking skeleton, Bach, Bolton, and Hendrickson on exploratory testing.

The artefact trail is free

Six months later, an engineer adding a feature asks "why does the auth flow take this path?" In a pipelined project, the answer is in an architecture decision record, not guessed from the code. An auditor reconstructing a trade-off finds the date and the alternatives that were considered. A successor maintainer inherits the project's record of its own thinking, not just its output.

The pipeline produces all of this as a side effect of producing the software — requirements with rationales, test cases traced to those requirements, decisions with alternatives, calibrated review reports, per-chunk handovers naming what was built and what deviated. Producing them as a by-product costs less than reconstructing them later.

What we got wrong

We assumed expert grounding would work everywhere. It doesn't. Expert grounding during review and design is demonstrably effective. During code construction, its effect is undemonstrated. The review phase, grounded in named experts, consistently caught issues that the construction phase failed to prevent. In design, the effect is visible in the output — architecture patterns in the code trace directly back to the named experts who were active during design. The expert shaped the design; the design shaped the code.

Worked example (now stale — kept because most teams are still in this situation)

On a project we built before GVM had the upstream gates that would catch this class of defect, the AI generated code that compiled and passed our tests against a mocked SDK. The function signature it was calling didn't exist on the real SDK — a hallucination. Tests against mocks couldn't see this; an integration test would have, but we hadn't written one yet. The expert-guided code review caught it before it shipped, by reading the code against the named SDK's published surface rather than against our own mocks. The general lesson: AI reviewers grounded in the right authority (the SDK documentation, an expert who knows the SDK) read code against external truth. Unguided AI reviewers read code against themselves.

The example is stale for current GVM. Today the external-boundary test (EBT) contract/collaboration distinction in /gvm-test-cases Phase 4 flags tests that mock external boundaries the wrong way, and the per-chunk integration smoke check in /gvm-build exercises the real boundary before the chunk is accepted — either gate would catch this defect upstream of code review. We keep the example because most teams doing AI-assisted development today have neither gate in place; the review layer is their only safety net, and it works for that.

The correction: construction is grounded in framework documentation — how to use the API correctly. Design and review are grounded in expert principles — whether the result meets quality standards. We include this because methodologies that aren't tested against real builds can't discover this class of failure. We tested ours. We found a failure. We corrected it. The released version already implements this separation.

The proof

We used GVM to build Assay, the tool that ran the experiment. Thirty-five hours from first commit to tagged release. 38 requirements, 207 passing tests, 7 rounds of AI code review, ten critical findings caught and fixed across the rounds, zero remaining at ship. The expert patterns are visible in the production code — Protocol-based abstractions because Ramalho and Fowler were active during the spec, atomic writes because Keeling was active during architecture decisions. You can read the code and trace each pattern to the expert who produced it.

The tool that was built to test the methodology was itself built by the methodology.

The cost only pays off against real downstream risk

GVM has a domain of fit. For a throwaway script or a fifty-line spike to test a hypothesis, the pipeline is overkill — vibe coding ships those faster than the pipeline can write the impact map. The trade is small upfront cost against large unpredictable downstream cost; that trade only pays off when the downstream cost is real. Run the full pipeline on every script and you waste effort. Skip it on a serious system and you take on risk it was designed to reduce. The user judges where the line falls.

And you need people

GVM doesn't replace roles. It shifts them from production to judgement. Business analysts still drive requirements — but instead of writing them from scratch, they're refining and approving what the AI drafted. Engineers still own the architecture — but instead of writing every line, they're making the design decisions and evaluating whether the generated code is right. QA still verifies — but against test cases the pipeline already traced to requirements. The same roles, operating at the speed of AI, spending their time on the decisions that require judgement while the AI handles the production at a speed they never could.
