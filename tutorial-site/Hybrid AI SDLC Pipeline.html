AI SDLC — Hybrid Pipeline

Spec-driven SDLC for AI implementers. One page. Everything you need.

Phase-driven rigor (for knowable domains) plus slice-driven iteration (for unknowable ones). 22 drop-in skills, 4 named subagents, a thin vault, and an always-on knowledge graph. Optimizes for learn fast, then build the right thing.

Overview

What is AI SDLC, and why it exists

Comprehensive upfront specification works when the domain is clear, the cost of rework is high, and compliance demands it. It falters when the spec itself is the unknown, when implementers are AI agents that benefit from plan-mode groundedness over 500-line sprint files, and when late discovery is more expensive than rework.

This pipeline keeps the strengths of phase-driven SDLC — vault, spikes, verification gates — and adds:

  • Risk-first slicing — build the scary thing first; slice 1 retires the most uncertainty, not whatever's easiest.
  • Two AI personas — Builder produces, Critic adversarially reviews. Same model, different prompts. Closes the bootstrapping problem of single-AI quality gates.
  • Three modes — Minimal / Standard / Heavy, picked by /triage. One pipeline shape doesn't fit every project.
  • 1-page mission brief + plan mode — discipline (gates, do-not-defer, ACs) lives in the brief; groundedness (real files, real code) lives in plan mode. Avoids the "500-line sprint file written against stale assumptions" failure.
  • Continuous validation — per slice, on a real device / user / data, not at project end.
  • Drift-check as pre-commit hook — vault rot is the #1 failure of spec-driven SDLC. A sub-second hook makes drift cheap to fix and expensive to ignore.
  • Thin vault — only what code can't carry: decisions, risks, slice memory. Components / contracts / schemas / test plans / screens live in code. Cure for spec rot at the root.
  • Project CLAUDE.md auto-generated — small (~20 lines), with the hard rule ("before editing code beyond trivial, check for active slice; if none, ASK"). Keeps Claude on the pipeline across sessions.
  • Graphify always-on — multi-modal knowledge graph (code + docs + papers + videos). PreToolUse hook nudges Claude toward GRAPH_REPORT.md before Glob/Grep; skills query the graph for reachability/blast-radius/path instead of re-scanning files.
  • milestone.md per slice — rolling state file (stage, next-action, on-resume data). Survives session death and context clears; /pulse reads it as primary source.
  • Honest Heavy mode — /heavy-architect produces only human-authored files (threat model, NFRs, requirements). Code-derived views (components / contracts / schemas) are regenerated on-demand by /sync before audits. Thin Vault preserved in all modes.

One sentence

Triage first, slice small, build the scary thing early, let Critic challenge the Builder, make reality validate the vault, reflect every slice.

Modes

The three modes

/triage opens every greenfield project; /adopt opens every brownfield one. Both pick the mode based on audience, compliance, team, and timeline. Mode tunes which steps run, how strict the gates are, and how thick the vault is.

ModeForPipeline shapeVault
Minimal Solo dev, MVP, exploration, one-off scripts Skip critique / user-test / drift-check. Lightweight slices. Thin (triage, concept, decisions, risks, slices)
Standard default B2C, small teams, product work All skills per slice. User-test before first UX-critical slice. Drift-check pre-commit. Thin
Heavy Compliance, regulated, audit-grade, public APIs Comprehensive upfront vault via /heavy-architect; mandatory critique + sign-off; reproducible validation; periodic /sync + /reduce. Comprehensive (threat-model, NFRs, requirements, etc. + code-derived views via /sync)
Default: Standard.

Upgrade to Heavy only when a specific regulation or external SLA forces it. Downgrade to Minimal only when the cost of the pipeline itself exceeds the project (one-off scripts, throwaway prototypes, personal tools).

Minimal mode

For solo developers building MVPs, exploring an idea, or shipping one-off scripts. Skips /critique, /user-test, /heavy-architect, /sync. The pipeline overhead is ~30 minutes total; the spike alone often saves a night.

/triage MINIMAL → /discover → /slice → /design-slice → /build-slice → /validate-slice → /reflect → loop

Standard mode (the default)

For B2C products, small teams, B2B SaaS, internal tools that other people will see. Adds the Critic, real-user testing for UX-critical slices, and pre-commit drift-check.

/triage STANDARD → /discover → (/risk-spike if HIGH risks) → (/user-test if B2C)
  → /slice → /design-slice → /critique → /build-slice → /validate-slice → /reflect → loop
  → maintenance: /drift-check (pre-commit), /reduce (when complexity grows), /critic-calibrate (every 10–20 slices)

Heavy mode

For HIPAA, PCI, SOC2, GDPR, regulated industries, public APIs with external SLAs. Adds upfront comprehensive architecture, mandatory Critic + human sign-off, reproducible validation with timestamps, periodic deep reconciliation.

/triage HEAVY → /discover (full role-play) → /risk-spike all → /heavy-architect
  → /user-test (if B2C)
  → /slice → /design-slice → /critique (mandatory + human sign-off)
  → /build-slice (compliance trail) → /validate-slice (signed) → /reflect → loop
  → maintenance: /drift-check (pre-commit), /sync (every 5–10 slices), /reduce (every 5 slices, mandatory)
Pipeline

The pipeline

Three flow shapes for three situations.

Greenfield (no code yet)

/triage                              ← decide mode, scope pipeline, build risk register
  ↓
/discover                            ← concept ↔ users ↔ constraints (one unified loop)
  ↓
/risk-spike                          ← validate HIGH risks with throwaway code on real env
  ↓
/user-test                           ← B2C only — mockup / prototype / slice with a real user
  ↓
[Heavy only: /heavy-architect produces upfront comprehensive vault]
  ↓
Per slice loop:
  /slice            → /design-slice → /critique
                              ↓
  /build-slice      → /validate-slice → /reflect → /commit-slice
  ↓ next slice

Continuous: /drift-check (pre-commit), /reduce (when complexity grows),
            /critic-calibrate (every 10–20 slices)

Brownfield (existing codebase)

Use /adopt instead of /triage. /adopt reads the code first (graphify scan + AST), treats README as hypothesis only, and optionally invokes /diagnose for forensic analysis. Then the same loop takes over.

/adopt [MINIMAL|STANDARD|HEAVY]            ← code is truth; docs are hypothesis
  ↓
/diagnose                              ← optional but recommended for >500 LOC. Produces diagnosis.html
  ↓ (repo owner annotates Confirmed: yes/no in browser, saves the HTML back)
/slice-candidates                      ← extracts confirmed findings → ordered backlog.md
  ↓
Same loop as greenfield, with backlog.md driving slice candidates.
Trust hierarchy in brownfield

1. Code observations (graphify AST + reachability). 2. User pain points + firsthand history. 3. Doc claims — treated as hypothesis, validated against code. README archaeology is dropped.

Bug fix (any mode)

/repro "<issue description>"          ← write a FAILING test that reproduces the bug. Confirm it fails.
  ↓                                     Add to shippability.md so it can never silently regress.
/slice "fix <issue>"                  ← AC = "make the failing test pass"
  ↓
Normal slice loop. Validation runs the shippability catalog
            — past slices' critical paths — to catch regressions.

Adding a feature to an existing AI SDLC project

Don't re-run /triage or /discover; the vault already exists. Jump to /slice "<feature>". If the feature touches a new auth model, external integration, or expensive decision: consider /risk-spike first.

The two-persona model

Every slice in Standard or Heavy mode uses two personas, same underlying model:

  • Builder (main thread): runs /slice, /design-slice, /build-slice, /validate-slice, /reflect.
  • Critic (spawned per /critique via Agent tool with subagent_type: "critique"): named subagent at ~/.claude/agents/critique.md with adversarial system prompt. Fresh context, no Builder reasoning trail. Returns blockers / majors / minors.

Cost: ~20% more tokens per slice. Cheaper than building the wrong thing. /critic-calibrate is the meta-skill that closes the feedback loop — it analyzes "Missed by Critic" entries across recent reflections and proposes targeted Critic prompt updates (human-reviewed; never auto-applied).

Mission brief vs sprint file

Mission brief (persistent, ~1 page)Plan mode (session)
Acceptance criteriaExact files to touch
Must-not-defer listTask sequence
Verification commandsCurrent-code-aware specifics
Vault / contract refsPlan visible to user for approval

Brief carries the discipline. Plan mode carries the groundedness. Avoids the "500-line sprint file written against stale assumptions" failure mode.

Skills

Skills catalog

22 drop-in Claude Code skills at skills/<name>/SKILL.md. Copy any directory into ~/.claude/skills/ and it works. Each skill has rich frontmatter so Claude triggers reliably on the right phrases.

Open the project

/triage all

Greenfield opener. Picks mode (Minimal/Standard/Heavy), builds risk register with reversibility tags, generates project CLAUDE.md (~20 lines, hard rule + vault discipline), offers graphify install. Pre-declarable: /triage MINIMAL, /triage STANDARD, /triage HEAVY.

Triggers: /triage, "start AI SDLC project", re-triage

/adopt all

Brownfield opener. Use INSTEAD of /triage for existing codebases. Reads code first (graphify), treats docs as hypothesis, offers /diagnose for forensic analysis, generates brownfield-aware CLAUDE.md with code-truth-over-docs rule.

Triggers: /adopt, "brownfield", "onboard existing codebase"

/diagnose all

Forensic, code-only audit. 11 analysis passes (intent, architecture, dead code, duplicates, half-wired features, contradictory assumptions, layering violations, AI-bloat signatures, oversized functions, dead config, test gaps). Produces a single self-contained diagnosis.html the repo owner annotates in their browser.

Triggers: /diagnose, "audit this repo", "forensic review"

/slice-candidates all

Reads annotated diagnosis.html, produces dependency-ordered backlog.md using graphify blast-radius and shared-evidence checks. Used by /adopt after the owner returns the annotated diagnosis.

Triggers: /slice-candidates, "build the backlog"

Discovery

/discover all

Concept + users + constraints in one unified loop. Three areas, one topic at a time. Output: enough understanding to identify the first slice candidate. Heavy mode adds full role-play per actor.

Triggers: /discover, "explore concept and users"

/risk-spike all

Validate risky third-party API assumptions with throwaway code on real environments — BEFORE any design. Step 2.5 spawns the field-recon subagent (fresh context, live WebSearch) to survey post-training-cutoff platform changes, quotas, deprecations. Returns a structured early-drop recommendation: if an official source contradicts the assumption, skip the empirical test and mark NO-GO. If docs confirm the assumption — empirical test still required (docs lie). Falls back gracefully when WebSearch is unavailable.

Triggers: /risk-spike, "validate assumption before design"

/user-test Standard / Heavy

Real-user validation gate — mockup, clickable prototype, or working slice tested with an actual person. Behavior over opinion ("show me how you'd…"). Captures Surprised / Ignored / Wanted / Stuck observations. Skip for backend, internal tools, or CLI-for-engineers.

Triggers: /user-test, /user-test mockup, /user-test prototype

/heavy-architect Heavy only

Upfront comprehensive vault for compliance / regulated projects. Produces only human-authored files (threat model with STRIDE per component, cost estimation at 1K/10K/100K, requirements, NFRs, diagrams, actor narratives). Code-derived views (components / contracts / schemas) are generated on-demand by /sync.

Triggers: /heavy-architect, "compliance architecture"

Slice loop

/slice all

Define the next thinnest valuable cut. Actively recommends candidates by risk-first ordering — gathers from risk register, recent deferrals, recent discoveries, unbuilt concept scope, aggregated lessons. Picks risk tier (low/medium/high) which gates whether /critique runs. Produces 1-page mission brief + milestone.md.

Triggers: /slice, "next slice", "what should we build next"

/design-slice all

Just-enough spec for THIS slice — not full architecture. Lists what's new vs reused, components touched (referencing code locations, not duplicating them), contracts added, ADRs locked. Tags every ADR with reversibility. ≤4 clarifying questions. Per-slice design, not whole-system; for Heavy-mode upfront vault, use /heavy-architect.

Triggers: /design-slice, "spec the current slice"

/critique Standard / Heavy

Adversarial review by separate Critic persona via Agent tool (subagent_type: "critique"). Attacks design along 9 dimensions, each grounded in a named expert framework — Wiegers (requirements), Fowler (refactoring smells), Newman (service contracts), OWASP + McGraw (security), Hendrickson / Bach / Bolton (edge case heuristics), Kiczales (cross-cutting concerns), and others. Citation-based grounding shifts the model from blended training-data heuristics to specific vetted methodology. Dimension 8 (web-known issues) uses live WebSearch. Dimension 9 (cross-cutting conformance) was added per CCC-1 / methodology-changelog v0.21.0 — operational/empirical evidence basis from cross-slice accumulation. Mandatory in Standard / Heavy. May skip low-tier slices that don't touch auth/contracts/data/multi-device/external integrations.

Triggers: /critique, "review the slice design"

/build-slice all

Plan mode + verification gates. Reads brief, enters plan mode to explore actual code (graphify-first), gets user approval, executes task-by-task with per-task verification. Mid-slice smoke at ~50%. Pre-finish gate: ACs pass, must-not-defer addressed, drift-check clean, no debug code. Updates milestone.md continuously for session-resume safety. Append-only build-log.md Events section is the flight recorder — write events before any tool call that could fail (screenshots, large file reads), so corrupted-binary errors and session deaths can't erase context.

Triggers: /build-slice, "ship the slice"

/validate-slice all

Reality check on real device / user / data. Per-AC PASS / FAIL with evidence (commands, screenshots, logs). Classifies failures: implementation bug / spec gap / reality surprise. Multi-instance required for sync/sharing/multi-user features. Runs the shippability catalog at pre-finish to catch regressions across past slices.

Triggers: /validate-slice, "reality check the slice"

/reflect all

Capture what reality taught you. Four categories: Validated / Corrected / Discovered / Deferred. Tracks Critic accuracy (VALIDATED / FALSE ALARM / NOT YET / Missed by Critic). Adds one critical-path test to shippability.md. Auto-archives the slice to slices/archive/ and refreshes _index.md. Previews next-slice candidates. The cure for spec rot.

Triggers: /reflect, "capture learnings", "slice retrospective"

Maintenance

/drift-check all

Detect-only vault-vs-code audit. Designed for pre-commit hooks (--fast: target <2s) and on-demand audits. Categorizes findings as DRIFT (blocker), UNSPECIFIED CODE, STALE CLAIM. --resolve walks each finding interactively. Skips slices/archive/.

Triggers: /drift-check, "is the vault still accurate"

/sync Heavy only

Bidirectional vault-code reconciliation. Regenerates code-derived files (components from AST, contracts from OpenAPI / Pydantic / type defs, schemas from data models) AND detects drift on the rest. Run periodically (every 5–10 slices) or before audits. Standard / Minimal use /drift-check instead.

Triggers: /sync, "reconcile architecture"

/reduce all

Complexity budget enforcer. Audits vault + code for over-engineering, speculative generality, dead claims, layer-cake. Mode-specific thresholds. Proposes a simplification slice. AI has no natural bias toward "less" — this is the counterweight. Mandatory every 5 slices in Heavy mode.

Triggers: /reduce, "complexity audit"

/archive all

Maintains slices/_index.md (the single lookup point) and slices/archive/. Convention: slices/ holds active only; completed slices auto-move on /reflect. --index-only rebuilds stale indexes after manual edits.

Triggers: /archive, "rebuild slice index"

/critic-calibrate all

Meta-skill. Analyzes "Missed by Critic" patterns across recent reflections, identifies blind spots, PROPOSES targeted updates to ~/.claude/agents/critique.md. Human reviews each proposal one at a time; never auto-applies. Run every 10–20 slices to close the Critic feedback loop.

Triggers: /critic-calibrate, "calibrate the Critic"

Utilities

/pulse all

Compact macro-state summary — mode, active slice stage + next action (read from milestone.md), risk exposure, regression health, Critic calibration status, top aggregated lessons, recommended next skill. --brief ~20 lines, default ~60, --full ~150. Read-only, never modifies. If the active slice is in build stage, also reads the tail of build-log.md Events — durable next-action signal that survives milestone.md staleness.

Triggers: /pulse, "where are we?", "pulse"

/repro all

Bug-fix discipline. Establishes a FAILING test that reproduces the issue, confirms it fails, adds to shippability.md. Hands off to /slice with the test as the slice's AC. Bug can never silently regress. Skip for typos / one-liners.

Triggers: /repro, "failing test first"

/commit-slice all

Generates an audit-grade commit message from the slice's vault artifacts (mission-brief / build-log / validation / new ADRs). Conventional-commit format with slice folder reference, AC count, Critic blockers addressed, ADR IDs, shippability entry. --merge (per BRANCH-1, methodology v0.35.0) commits on the current slice branch + no-ff merges back to the resolved default branch + safe-deletes the slice branch (explicit confirmation at each checkpoint).

Triggers: /commit-slice, "audit commit"

Named subagents

Four skills delegate to named subagents at ~/.claude/agents/<name>.md. The agent file carries the load-bearing prompt; the skill orchestrates inputs/outputs.

AgentInvoked byWhy fresh context
critique/critiqueAdversarial isolation — Critic must NOT see the Builder's reasoning trail. Uses citation-based expert grounding (Wiegers, Fowler, Newman, OWASP, etc.) per dimension.
critic-calibrate/critic-calibrateObjective pattern analysis across past reflections; benefits from a clean read.
field-recon/risk-spike Step 2.5Live WebSearch survey for post-cutoff platform changes, quotas, deprecations. Web search noise stays out of the main thread; returns findings + structured early-drop recommendation. Asymmetric rule: drop on authoritative contradiction, never on confirmation.
diagnose-narrator/diagnose Step 6.5Engaging executive summary for diagnosis HTML; doesn't pollute the main /diagnose conversation.

Fork-friendly skills

Three skills can run as forks (CLAUDE_CODE_FORK_SUBAGENT=1) — they inherit parent context, run in the background, report when done.

  • /risk-spike — parallel risk validation: /fork /risk-spike R1, /fork /risk-spike R2, /fork /risk-spike R3 in one turn.
  • /reduce — long-running complexity audit on larger projects; user keeps moving.
  • /sync — Heavy mode 1–3 min reconciliation off the main thread.

Without the env var, these run in the main thread with the same outputs — just blocking.

Vault

Vault structure (thin vault)

The vault holds only what code can't carry: decisions, rationale, risks, slice memory. Components / contracts / schemas / test plans / screens live in code (or auto-derived views via OpenAPI, type defs, AST queries). Same fact in two places drifts; fewer places = less drift.

Reframe: think of the vault as the AI's project memory across sessions, not the project's documentation for new readers. AI memory needs decisions made (so it doesn't relitigate), risks (so it doesn't re-discover), slice history (so it knows what's done), reflections (so it learns). It does NOT need every component, every endpoint, every test — those are derivable from code.

Standard / Minimal mode (the default)

architecture/
  CLAUDE.md                  # auto-generated by /triage; pipeline enforcement
  triage.md                  # mode + rationale + initial risks
  concept.md                 # what + who + constraints (one file; no separate actors/)
  risk-register.md           # risks + spike status + reversibility tags
  decisions/                 # ADRs — the WHY of expensive decisions
    ADR-001-*.md             #   each tagged: reversibility: cheap | expensive | irreversible
  spikes/                    # from /risk-spike
    spike-*.md
  slices/                    # THE CORE
    _index.md                # the lookup: active list + recent-10 + aggregated lessons + pointer to archive catalog
    slice-NNN-<name>/        # ACTIVE slices only (no reflection.md yet)
      milestone.md           # rolling state file (updated by every skill); /pulse reads this first
      mission-brief.md       # per-slice intent + ACs + risk-tier + critic-required + gates
      design.md              # just-enough — references code locations, doesn't duplicate them
      critique.md            # Critic's review + Builder's responses (may be "skipped" for low-risk slices)
      build-log.md           # ## Events (append-only flight recorder) + ## Summary (filled at slice end)
      validation.md          # reality-check per AC + shippability catalog results
      (reflection.md appears → /reflect auto-archives the slice)
    archive/                 # ALL completed slices (auto-moved by /reflect)
      _index.md              # full chronological catalog
      slice-001-<name>/      # frozen history (milestone.md frozen as "complete")
      slice-002-<name>/
      ...
  user-tests/                # from /user-test (B2C only)
    <test-name>.md
  shippability.md            # regression catalog — one critical-path test per slice
  drift-log.md               # drift events over time
  lessons-learned.md         # accumulated insights, chronological
  critic-calibration-log.md  # audit trail of /critic-calibrate runs
  changelog.md               # pipeline-bypass log, if any

graphify-out/                # query cache — multi-modal knowledge graph
  graph.json                 # AST + semantic + INFERRED edges over code, docs, papers, videos
  GRAPH_REPORT.md            # one-page digest: god nodes, communities, surprising connections

Deliberately NOT in the vault (Standard / Minimal)

  • components/ — read the code; or $PY -m graphify code for a graph view.
  • contracts/ — auto-generated from OpenAPI / Pydantic / type defs.
  • schemas/ — from data models / migrations.
  • test-plan/ — tests are the plan.
  • frontend/ — code structure shows screens; mission briefs reference them by path.
  • actors/ — inline section in concept.md (one paragraph per actor); no separate files.

If you find yourself wanting one of these, ask: would code (or a code-derived view) capture this better? Usually yes.

Reversibility tags on every ADR

Each ADR carries a frontmatter reversibility: field. Cheap-to-change → lock fast. Expensive-to-change → lock late, after evidence.

DecisionReversibilityWhen to lock

Logging formatcheapFirst slice that logs
Framework choice (React vs Vue)expensiveAfter /discover confirms team fit
Database choice (Postgres vs Mongo)expensiveAfter /risk-spike if any data model is novel
API contract (versioned public API)expensiveAfter 2+ slices exercise the endpoints

Slice folder shape

Every active slice folder contains a sequence of artifacts produced by the loop. milestone.md is the rolling state file updated by every skill; /pulse reads it as the primary source.

Graphify

Graphify integration — the always-on layer

Graphify turns the codebase + vault + external references into a queryable multi-modal knowledge graph. Code (25 languages via tree-sitter), docs (markdown / HTML / RST), Office files (.docx / .xlsx), papers (PDF with citation mining), images (vision models), video / audio (transcribed locally with faster-whisper). Result: sub-second structural queries (reachability, blast-radius, shortest path) against a pre-built graph instead of re-scanning source.

Installed by /triage and /adopt

Both project openers offer (recommended yes):

# Pick the right Python path for the platform
case "$(uname -s)" in
  MINGW*|MSYS*|CYGWIN*) PY="$HOME/.claude/.venv/Scripts/python.exe" ;;
  *)                    PY="$HOME/.claude/.venv/bin/python" ;;
esac

$PY -m graphify install                # base skill at ~/.claude/skills/graphify/
$PY -m graphify claude install         # PreToolUse hook — every Glob/Grep redirects through GRAPH_REPORT.md
$PY -m graphify hook install           # post-commit + post-checkout git hooks for auto-rebuild
$PY -m graphify .                      # initial graph build
git init is the user's call

If graphify hook install fails because there's no git repo, /triage stops and asks. Initializing a repo touches identity (default branch, user.name/email, authorship, CI assumptions) — not Claude's to do silently.

The always-on pattern

  1. User asks: "How does payment work?"
  2. Claude's Glob/Grep would normally scan files.
  3. PreToolUse hook intercepts: "Knowledge graph exists. Read GRAPH_REPORT.md first."
  4. Claude reads GRAPH_REPORT.md (one file, ~2k tokens) instead of scanning raw files (~100k+ tokens).
  5. Finds the "payment" community in the graph, the god nodes, the surprising connections.
  6. Answers with structure-aware reasoning.

This happens transparently every session. Nothing for the user to remember.

Semantic archive retrieval — the slice-108 problem

As projects cross ~30 slices, _index.md's "Aggregated lessons" + "Recent 10" only capture nearby history. A lesson from slice-008 can be relevant when designing slice-108 but is no longer in the index.

Graphify solves this without new tooling — archived reflections are already in the graph. Query semantically:

# When designing slice-108 that touches image handling
$PY -m graphify query "past lessons about image handling"
# → surfaces slice-023's EXIF orientation lesson, slice-045's HEIC edge cases, etc.

# Before spiking a new integration
$PY -m graphify query "past spikes with OAuth scopes"

# When critiquing a design touching auth
$PY -m graphify query "past misses in auth-related slices"

Bottom line: _index.md = recent context (in-context, free); graphify query = deep archive retrieval (~1.7k tokens per query).

Edge confidence + rationale comments

Graph edges are tagged: EXTRACTED (found in source, confidence 1.0), INFERRED (with confidence score), AMBIGUOUS (flagged for review). The Critic uses this — INFERRED edges with low confidence = design assumptions to challenge.

Rationale comments in code (# WHY:, # NOTE:, # IMPORTANT:, # HACK:) become graph edges alongside vault ADRs. The "why" lives in both places and is queryable in one.

Patterns

Patterns & anti-patterns

Healthy patterns

  • Slice 1 retires the scariest risk. Cross-device sync, novel auth, unproven API, new UX. If it works, everything else is execution. If it doesn't, you pivot cheaply.
  • Verb-object slice names. add-receipt-upload, enable-sync, fix-thumbnail-orientation. Never phase-2, slice-N, or vague nouns.
  • ≤5 acceptance criteria, ≤1 day per slice. If a slice would take longer, split. If splitting produces a slice with no user-visible value, the original "feature" was probably 2+ features.
  • Must-not-defer list every slice. Even trivial slices: auth, validation, error paths, observability, authorization — these can't be TODOs.
  • Mid-slice smoke at ~50%. Hit the endpoint, open the page, install on the device. Catches "builds but doesn't work" before more is built on a broken base.
  • Real-environment validation. Tests passing ≠ feature works. Real device, real user, real data — especially for HEIC files, EXIF orientation, weird unicode, multi-instance.
  • Reflection captures honestly. Validated / Corrected / Discovered / Deferred + Critic calibration. If "Discovered" is empty for several slices: either you're not learning or not capturing.
  • One critical-path test per slice. /reflect adds it to shippability.md. /validate-slice runs the full catalog at pre-finish — past slices' critical paths catch regressions introduced by the current one.
  • Tests-first for bug fixes. Run /repro to write a failing test before any fix code. Bug can never silently regress.
  • ADRs are append-only. Supersede with a new ADR (supersedes: + superseded-by: frontmatter); never edit in place. Decision history is audit trail.
  • Reversibility tags on every ADR. Lock cheap decisions early. Defer expensive ones until a slice actually needs them. Defer irreversible ones until /risk-spike validates.

Anti-patterns

Auto-creating component files in Standard mode

Symptom: /design-slice produces components/X.md and contracts/Y.md in a Standard-mode project.

Why bad: Thin vault is the Standard default. Component files duplicate what code already shows; they drift; they're maintenance burden.

Fix: tell Claude "we're in Standard mode (thin vault). The design.md should reference code locations, not create separate component files." Verify architecture/triage.md shows mode: Standard.

Skipping /critique because "the design is obvious"

Symptom: User wants to skip critique to save time on a "trivial" slice.

Why bad: The "trivial" ones are where you stop watching. Critic catches the auth gap on the trivial endpoint that exposes data.

Fix: only skip in Minimal mode AND for slices with no new auth, no new contracts, no new external dependencies. When in doubt: run it. ~5 minutes of Critic vs hours of bug.

Letting Claude bypass the pipeline without confirming

Symptom: User asks Claude to "fix this bug" or "add this feature" and Claude jumps to editing code without checking for an active slice.

Why bad: This is the "ditch the pipeline" failure. Discipline collapses; vault rots.

Fix: the auto-generated ./CLAUDE.md (~20 lines) carries the hard rule. Big CLAUDE.md gets skimmed — keep it small. If Claude still bypasses: explicitly ask "did you check for an active slice?"

Treating the vault as documentation for new humans

Symptom: Vault grows comprehensive component descriptions, architectural diagrams, "system overview" docs for hypothetical new readers.

Why bad: That's not what the vault is for in Standard mode. It's the AI's project memory — decisions, risks, slice history. Documentation for humans is a different artifact (and probably best generated on demand from code).

Fix: review what's in the vault. If it's "what code does," delete it. If it's "why we decided X," keep it.

Running /heavy-architect in Standard mode

Symptom: User wants "a real architecture" upfront, runs /heavy-architect in a Standard project.

Why bad: Produces 30+ files of comprehensive vault that will drift, won't be maintained, and weren't needed.

Fix: /heavy-architect refuses if mode isn't Heavy. If you legitimately need it: /triage --re-triage to Heavy first.

One slice for "the whole feature"

Symptom: A slice with 12 acceptance criteria and a 3-day estimate.

Why bad: Defeats the slice model. Mid-slice smoke gates won't work; validation becomes fuzzy; reflection becomes "we built a lot."

Fix: split. Each slice ≤5 ACs, ≤1 day. If splitting produces a slice with no user-visible value, the feature was probably 3 features mashed together.

Reflection as victory lap

Symptom: Reflections that say "all ACs passed, design was correct, no issues."

Why bad: Teaches nothing. Next slice doesn't benefit. Critic calibration data isn't generated.

Fix: enforce honesty discipline. Reflections must populate Validated / Corrected / Discovered / Deferred sections explicitly. If Discovered is empty across multiple slices: push harder.

Drift-check disabled because "it's annoying"

Symptom: Pre-commit hook disabled or bypassed with --no-verify.

Why bad: Drift compounds silently. Six months later the vault claims sendgrid; reality is resend.

Fix: if the hook is too slow, use --fast mode (target <1s on thin vault). If too noisy, the vault probably has bad assumptions about code paths — fix the vault, not the hook.

Trusting README in brownfield

Symptom: /adopt writes concept.md from what README claims.

Why bad: README is aspirational at best, fictional at worst. Code is truth.

Fix: trust hierarchy is graphify scan first → user firsthand history second → docs third (as hypothesis only). Run /diagnose for non-trivial brownfields to confirm reality.

Doc archaeology for ADRs

Symptom: Reverse-engineering historical ADRs from existing docs the user can't articulate firsthand.

Why bad: Better 0 ADRs than 5 fictional ones.

Fix: /adopt only captures ADRs the user can describe themselves — context, options considered, why this won. If user struggles to articulate: skip.

Worked examples

Worked examples

Minimal mode — personal CLI tool (duplicate-photo finder)

Solo, weekend project, no users besides you. Total time ≈ 3–4 hours including spike.

Step 1 — Triage. /triage "CLI tool to find duplicate photos and emit a CSV report"
Claude: 5 questions, one at a time. Mode: Minimal (solo, MVP, one-off). Risk register: | R1 | Perceptual hash accuracy on similar-but-different photos | irreversible-if-wrong | YES | Pipeline: /risk-spike R1 → /discover → /slice → ... → /reflect → loop Wrote: triage.md, risk-register.md, ./CLAUDE.md (~20 lines) Run /risk-spike R1 next.
Step 2 — Risk-spike. /risk-spike R1 — 15-line script tests imagehash on 20 known pairs.
Result: - True duplicates: distance ≤ 5 (10/10 detected) - Similar-but-different: distance 8–25 (separable) - Threshold of 6 cleanly separates the two. Decision: GO. Caveat: expose threshold as a CLI flag. Wrote: spikes/spike-phash.md Updated: risk-register.md (R1 → RETIRED)
Step 3 — Skip /discover (Minimal, trivial), straight to /slice.
Step 4 — /slice "scan a folder and write CSV of duplicates" — 5 ACs, must-not-defer (handle unreadable files, show progress on ≥100 files), out of scope (GUI, auto-deletion, cloud sync).
Step 5 — /design-slice (lightweight in Minimal). One ADR: ADR-001 use imagehash.phash, reversibility: cheap.
Step 6 — /build-slice → /validate-slice → /reflect. Critique skipped (Minimal, trivial). Real test on 50 photos with 5 known clusters → all 5 detected.

What the pipeline added: ~30 minutes of overhead vs "just write the script". The spike alone saved a night of fighting hash thresholds.

Standard mode — household expense tracker (B2C web app)

Two users per household, subscription tier later, ~3 months solo. Total time ≈ 4–6 hours per slice.

Step 1 — Triage. Mode: Standard. Three risks identified:
  • R1 Cross-user data sharing model — irreversible — spike YES
  • R2 Receipt OCR accuracy at launch — expensive — try Tesseract on 30 samples
  • R3 Subscription billing integration choice — expensive — defer until needed
Step 2 — Spike R1 + R2. R1 → "household" entity with linked accounts is cleaner. GO. R2 → 60% extract amount correctly, 40% wrong (handwritten, faded ink). NOT good enough as primary entry method. NO-GO for OCR-as-default; CONDITIONAL for OCR-as-assist (user reviews).
Step 3 — /discover. WHAT (concept), WHO (household member + admin, inline in concept.md — thin vault), CONSTRAINTS (Next.js + Postgres on Vercel + Supabase, captured as ADRs).
Step 4 — /user-test mockup for first slice (household creation + invitation). Findings:
  • Surprised: User expected "Create household" to ask for household NAME. → Add name field.
  • Ignored: Long welcome paragraph. → Cut it.
  • Wanted: A way to undo invitation. → Add cancel-invitation in slice 2.
  • Stuck: Confused about "joining" vs "creating" — verb mattered. → Clearer button labels.
Step 5 — Slice 1 loop. /slice → /design-slice → /critique → /build-slice → /validate-slice → /reflect.
Critic finds: - B1: ADR-004 says invitation tokens have 24h expiry, but no DB cleanup. → expired tokens accumulate. - M1: Re-invitation flow not specified (what if user invites same email twice?). - m1: "Household name" field has no length limit. Builder addresses B1 (DB cleanup task), M1 (re-invitation cancels previous), defers m1.
Step 6 — /validate-slice: real test, two browser windows, two emails, full invitation flow. PASS — all 5 ACs verified with screenshots.
Step 7 — /reflect.
Validated: invitation flow works for normal case; magic links work. Corrected: design.md said magic link via Vercel KV; reality used Supabase Edge Functions (cleaner). Updated. Discovered: re-invitation case wasn't in mission brief, but Critic caught it (M1). Critic calibration: B1 VALIDATED (token cleanup hit mid-build). M1 VALIDATED (came up in real test). m1 NOT YET (deferred). Lesson added: in invitation/sharing slices, always think about the re-invite case.

What Standard adds vs Minimal: /critique every slice (~30 min), /user-test before first UX-critical slice, stricter mission briefs, pre-commit /drift-check. Cost: real. Benefit: post-launch bugs avoided.

Principles

Five principles

Every skill traces back to one or more.

1Iterate per slice, not per phase

Why: You learn what to build by building. Phase-driven pipelines lock design decisions before evidence exists. By validation time, the wrong thing is built at scale.

How: /slice defines the next cut. /design-slice scopes the spec to that cut only. /build-slice ships it. /reflect captures what reality taught you. Each cycle = working code + updated spec.

Counter-principle: for compliance-driven work where contracts must be defined upfront (FDA, HIPAA, banking), Heavy mode relaxes this — contracts frozen before any slice runs. Even then, slices remain the execution unit.

2Risk-first ordering

Why: Projects die from unknown-unknowns discovered late. Build the scary thing first — novel auth, unproven API, new UX, cross-device sync. If it works, everything else is execution. If it doesn't, you pivot cheaply.

How: /triage builds the risk register. /slice selects the next cut by risk, not dependency order. /risk-spike runs before design when validation is cheaper than implementation (a 30-min spike beats an 8-sprint rebuild).

3Two AI personas

Why: Same AI on both sides of a quality gate is the bootstrapping problem. AI writes a spec → same AI validates → same AI builds. Errors propagate silently because validator shares the writer's blind spots.

How: /critique invokes a named subagent at ~/.claude/agents/critique.md with the load-bearing adversarial prompt — stance, 8 attack dimensions each grounded in a named expert framework (Wiegers for requirements, Fowler for refactoring smells, Newman for service contracts, OWASP for security, Hendrickson / Bach / Bolton for edge case heuristics, etc.), specificity / honesty rules, output format. Citation-based grounding shifts the model from blended training-data heuristics to specific vetted methodology. Fresh context — no Builder reasoning trail. /critic-calibrate is the meta-skill that mines past misses to propose Critic prompt improvements (human-reviewed, never auto-applied).

Cost: ~20% more tokens per slice. Cheaper than building the wrong thing. Failure mode to watch: Critic becomes a rubber stamp if its prompt is too mild.

4Reversibility-tagged decisions

Why: Not all decisions are equal. Locking a UI color is free; locking a database schema or API contract is a migration. Phase-driven pipelines treat all decisions the same, forcing early commitment on expensive things before evidence supports them.

How: Every ADR has a reversibility: frontmatter field (cheap / expensive / irreversible). /design-slice only locks expensive decisions when the slice needs them; cheap ones lock early to unblock work.

5Thin vault — AI memory, not system documentation

Why: The biggest failure mode of spec-driven SDLC is spec rot. The deeper cause: trying to document what code already shows. Component descriptions, contract schemas, test plans drift not because discipline is weak, but because the same fact lives in two places (code + vault) that want to diverge. The cure is fewer places, not more discipline.

The split: code is the source of truth for "what" (components, contracts, data shapes, test coverage, frontend structure). Vault is the source of truth for "why" (decisions, rationale, risks, intent, learnings). Reframe: vault is the AI's project memory across sessions, not project documentation for new readers.

Heavy mode behavior: the same principle holds. /heavy-architect produces only human-authored files (threat model, NFRs, requirements, diagrams, actor narratives). Code-derived views (components / contracts / schemas) are regenerated on-demand by /sync right before audits — not maintained between runs. Audit artifacts when you need them; no drift between audits.

Graph as structural index: /triage and /adopt install graphify — multi-modal knowledge graph over code + vault + external references. Skills query the graph for reachability, blast-radius, and shortest path instead of scanning files. Rationale comments in code (# WHY:, # NOTE:) become graph edges alongside vault ADRs — the "why" is queryable in one place.

What these principles reject

  • Waterfall-as-default. Phase-driven SDLC encodes a specific theory (specs come first, exhaustively) and applies it uniformly. Treated here as one mode (Heavy), not the default.
  • Single-AI quality gates. One AI writing and another part of the same AI validating is theater, not review.
  • Artifact completeness as a goal. A vault with 200 files and stale claims is worse than one with 20 files that match reality.
  • Phase gates as milestones. The only milestone is working software a real user / device has touched.

What these principles are honest about

  • Slice-driven has its own failure modes: losing sight of the whole, building inconsistently across slices, accumulating tech debt. /reduce and /drift-check are the counterweights, but discipline is still required.
  • Two-persona costs tokens, and the Critic isn't perfect. Calibration happens over time by tracking /reflect's "Missed by Critic" entries; /critic-calibrate closes the loop.
  • Heavy mode exists because these principles don't always fit. Compliance domains genuinely need upfront contracts. Don't force slice-driven where it's wrong.
Setup

Install (one-time per machine)

If you're using this pipeline, you already have Claude Code. Use it to install:

How to invoke

cd into the AI SDLC source directory, open Claude Code, and say:

"Install this. Read INSTALL.md and follow it."

Claude reads INSTALL.md, detects what's on your machine, asks before mutating shared files (global CLAUDE.md, settings.json), executes the install, and verifies at the end. Idempotent — safe to re-run if already partially configured.

What the install does

  1. Verifies / creates ~/.claude/.venv (Python 3.11+).
  2. Installs graphify (editable from ~/.claude/packages/graphify if present, else PyPI).
  3. Registers the graphify Claude Code skill at ~/.claude/skills/graphify/.
  4. Adds the PY=... shared-Python convention to ~/.claude/CLAUDE.md (with your confirmation).
  5. Adds CLAUDE_CODE_FORK_SUBAGENT=1 to ~/.claude/settings.json (so /fork works).
  6. Copies AI SDLC skills + agents into ~/.claude/skills/ and ~/.claude/agents/.

After setup: restart Claude Code so the env var loads. Then /triage (greenfield) or /adopt (brownfield) runs a preflight check on first use; if anything is missing, it fails fast and points you back to INSTALL.md.

The shared Python convention

Don't activate the venv. Call the interpreter by absolute path. Cross-platform pattern (the same one used in /triage, /adopt, /diagnose, /slice-candidates):

case "$(uname -s)" in
  MINGW*|MSYS*|CYGWIN*) PY="$HOME/.claude/.venv/Scripts/python.exe" ;;
  *)                    PY="$HOME/.claude/.venv/bin/python" ;;
esac

$PY -m graphify .          # build initial graph
$PY -m graphify query "what does receipt-service depend on?"
